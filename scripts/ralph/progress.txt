## 2026-04-08 - US-001: Fix feature store numeric overflow for extreme feature values

- **Root cause:** `feature_store.feature_value` column was `numeric(18,8)` (max ~9.99e9). NVDA's OBV (On-Balance Volume) cumulative sum reached ~35.6 billion over 5 years, overflowing the column type.
- **Fix 1 (schema):** Directly widened column: `ALTER TABLE feature_store ALTER COLUMN feature_value TYPE numeric(24,8)` â€” applied to running cluster via kubectl exec on 2026-04-08.
- **Fix 2 (code):** Added `_FEATURE_VALUE_CLAMP = 9.9e15` constant and clamping logic in `write_features()` in `ml/features/store.py` to log a warning and clamp any value that would overflow even the new wider column.
- **Fix 3 (migration):** Created `ml/alembic/versions/0002_widen_feature_store_numeric.py` to document the schema change for future fresh deployments.
- **Verified:** NVDA test pod completed with `Wrote 48576 feature rows for NVDA` â€” no NumericValueOutOfRange error.
- **Files modified:** `ml/features/store.py`, `ml/alembic/versions/0002_widen_feature_store_numeric.py`
- **Learnings for future iterations:**
  - Cumulative volume indicators (OBV, AD-Line) accumulate billions of shares for high-volume tickers; always use at least `numeric(24,8)` for feature values.
  - The ML alembic runner has no `alembic.ini` â€” migrations must be applied manually via `psql` or a migration job until a proper runner is configured.
