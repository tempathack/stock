Service Modules (14 total)

yahoo_finance.py

Fetches OHLCV data from Yahoo Finance API with Stooq fallback. Implements exponential backoff (3 retries), batch ticker processing, and market calendar awareness. Produces structured DataFrames ready for Kafka.

Expansion: Add alternative data sources (Alpha Vantage, Polygon.io), support crypto/forex feeds, add news sentiment data.

kafka_producer.py

Confluent Kafka producer publishing OHLCV records to intraday-data and historical-data topics. Handles serialization, delivery callbacks, and error reporting.

Expansion: Add schema registry (Avro), dead letter queue, exactly-once semantics with idempotent producer.

prediction_service.py

Orchestrates model inference: tries KServe first, falls back to local file-based model. Redis-caches predictions (1h TTL). Supports multi-horizon (1d/7d/30d), bulk predictions, and A/B routing.

Expansion: Add ensemble averaging across horizons, confidence calibration, prediction explanation caching.

kserve_client.py

Async HTTP client for KServe V2 inference protocol. Sends feature vectors, parses V2 response format. Configurable timeout (30s default), graceful degradation on failure.

Expansion: Add gRPC support, model warm-up requests, batch inference optimization.

market_service.py

Market data aggregation from TimescaleDB. Computes technical indicators (RSI, MACD, Bollinger, ATR, ADX, VWAP, OBV, returns) on-demand from continuous aggregates.

Expansion: Pre-compute indicators via materialized views, add sector rotation analysis, market regime detection.

feast_service.py

Queries Feast SQL registry for FeatureView metadata and last_updated timestamps. Monitors feature staleness to alert when materialization jobs fail.

Expansion: Add feature lineage tracking, data quality metrics, feature importance correlation.

health_service.py

Deep health checks: DB connectivity, Redis ping, model freshness validation, component staleness detection. Returns structured health response for K8s probes.

backtest_service.py

Historical model validation: loads cached predictions, compares against actual prices, computes RMSE, MAE, MAPE, directional accuracy over configurable windows.

ab_service.py

A/B test infrastructure: routes requests to primary vs. canary model based on traffic weights, logs assignments, tracks per-model performance for comparison.

flink_service.py

Queries Flink REST API for active job statuses across all three deployments. Results cached (10s TTL) to avoid overwhelming Flink JobManager.

kafka_lag_service.py

Computes per-partition consumer lag for the processed-features topic using Kafka Admin API. Cached at 15s intervals.

price_feed.py

WebSocket background task that polls Yahoo Finance for live prices during market hours. Broadcasts to all connected clients. Market-status aware (pauses outside trading hours).

7. KServe Model Serving

KServe provides production-grade model serving with auto-scaling, canary deployments for A/B testing, and the V2 inference protocol. Models are loaded from MinIO (S3-compatible) and served via MLServer with the sklearn runtime.

Primary InferenceService

Name: stock-model-serving. Loads the active winner model from s3://model-artifacts/serving/active. Auto-scales 1-3 replicas based on load. Resources: 500m-2000m CPU, 1-4Gi RAM.

Endpoint: POST /v2/models/stock-model-serving/infer

Canary InferenceService

Name: stock-model-serving-canary. Loads candidate model from s3://model-artifacts/serving/canary. Scales to 0 when not in use. Used for A/B testing new models before full promotion.

Endpoint: POST /v2/models/stock-model-serving-canary/infer

Custom Serving Runtime

stock-sklearn-mlserver using MLServer 1.6.1 with mlserver_sklearn.SKLearnModel implementation. Supports V2 protocol on HTTP (:8080) and gRPC (:8081).

Certificate Management

Cert-Manager handles TLS certificates for KServe webhook endpoints, ensuring secure internal communication.

9. Apache Flink (Stream Processing)

Three PyFlink 1.19 jobs process streaming data in real-time with EXACTLY_ONCE semantics, RocksDB state backend, and checkpoints persisted to MinIO. Each runs as a FlinkDeployment CRD managed by the Flink Kubernetes Operator.

OHLCV Normalizer

Input: Kafka intraday-data

Processing: Validates and normalizes OHLCV records (price sanity checks, volume normalization, timestamp alignment).

Output: JDBC sink to PostgreSQL ohlcv_intraday hypertable (PRIMARY KEY upsert).

Shows: Clean, validated intraday data flowing into TimescaleDB for downstream aggregation and ML training.

Interacts with: Kafka (source), PostgreSQL (sink), MinIO (checkpoints).

Indicator Stream

Input: Kafka intraday-data

Processing: HOP windows (5-minute slide, 20-minute size) computing EMA-20, RSI-14, MACD signal line using custom UDAFs.

Output: Kafka processed-features topic.

Shows: Real-time technical indicator computation as market data streams in, enabling low-latency feature serving.

Interacts with: Kafka (source + sink), MinIO (checkpoints).

Feast Writer

Input: Kafka processed-features

Processing: Parses feature batches, constructs DataFrames, pushes to Feast online store.

Output: Redis (Feast online store) via PushMode.ONLINE.

Shows: Feature vectors available in Redis within seconds of market data arriving, enabling real-time inference.

Interacts with: Kafka (source), Redis/Feast (sink), MinIO (checkpoints).

10. Kafka Consumer (Batch Writer)

A dedicated Python service that consumes Kafka messages in batches and writes them to PostgreSQL hypertables. Runs 2 replicas for load distribution across partitions.

📦
Kafka Consumer Service
services/kafka-consumer/ | 2 replicas | Batch: 100 rows / 5s timeout

Consumes from intraday-data and historical-data topics. Batches 100 records or flushes every 5 seconds (whichever comes first). Performs upsert operations to ohlcv_daily and ohlcv_intraday hypertables with conflict resolution on (ticker, date/timestamp).

Modules: main.py (consumer loop, signal handling), processor.py (message batching), db_writer.py (DB upsert), metrics.py (Prometheus @ :9090), config.py (env-based config), logging.py (structured logs).

Expansion: Add Kafka Streams for richer processing, implement exactly-once with transactional consumer, add DLQ for failed records, support Avro deserialization.

16. Drift Detection & Auto-Retraining

Daily drift detection runs at 10 PM UTC on weekdays, checking for data drift, concept drift, and prediction drift. When drift exceeds thresholds, automatic retraining is triggered.

DataDriftDetector

Compares recent feature distributions against the training baseline using KS-test (p-value < 0.01 threshold) and PSI (Population Stability Index > 0.2 threshold). Reports severity based on % of features drifted.

ConceptDriftDetector

Monitors the relationship between features and target variable. Detects when the underlying pattern (e.g., which features predict returns) changes, even if individual feature distributions remain stable.

PredictionDriftDetector

Compares prediction distribution against recent actuals. High prediction drift without data drift suggests model degradation.

Auto-Retraining Trigger

When drift exceeds thresholds, the trigger module launches a full training pipeline with --skip-shap for faster turnaround. Logs all decisions to drift_logs table.
