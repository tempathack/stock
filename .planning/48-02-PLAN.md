# 48-02-PLAN — Deep Health Checks

## Goal
Comprehensive health checking with degraded-state detection and a /health/deep endpoint.

## Changes

### 1. `services/api/app/services/health_service.py` (NEW)
- `check_db()` — Execute `SELECT 1` via async session; return ok/error
- `check_kafka()` — Attempt Kafka metadata list_topics with 5s timeout; return ok/error
- `check_model_freshness()` — Query model_registry for latest updated_at; stale if >7 days
- `check_prediction_staleness()` — Query predictions for latest prediction_date; stale if >24h
- `run_deep_checks()` — Run all 4 checks, compute overall status ("ok" | "degraded" | "unhealthy")

### 2. `services/api/app/routers/health.py`
- Keep existing `GET /health` but enhance with deep check integration
- Add `GET /health/deep` for comprehensive component-level report
- Status logic: all ok → "ok"; any warning → "degraded"; db down → "unhealthy"
- Pydantic response models for component checks

### 3. `services/api/app/config.py`
- Add `MODEL_FRESHNESS_THRESHOLD_DAYS: int = 7`
- Add `PREDICTION_STALENESS_HOURS: int = 24`

### 4. Unit tests: `services/api/tests/test_health_deep.py`
- Test /health returns degraded when DB unreachable
- Test /health/deep returns all 4 component statuses
- Test model freshness and prediction staleness thresholds

## Success Criteria
- GET /health returns {status: "degraded"} when DB is unreachable
- Health check validates: DB connectivity, Kafka reachability, model freshness (<7d), prediction staleness (<24h)
