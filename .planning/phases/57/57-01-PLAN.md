## Plan 57-01 — Remove Legacy Manifests, Update deploy-all.sh, Reclaim PVC

### Objective
Delete the legacy PVC-based model-serving Deployment/Service manifests that were kept as rollback safety nets in Phases 55–56, update `deploy-all.sh` to stop applying them and actively clean up existing clusters, and confirm the PVC is repurposed (not deleted).

### Requirements Addressed
- **KSERV-13**: Legacy `model-serving.yaml` Deployment removed
- **KSERV-14**: `deploy-all.sh` updated with MinIO + KServe deployment steps (cleanup of legacy references)

### Why
Phase 55 scaled the legacy model-serving Deployment to `replicas: 0` and annotated both manifests with "Delete in Phase 57." Phase 56 completed the API-side migration to KServe V2 inference protocol. The legacy manifests are now dead code:
- They are never scaled above 0
- The API no longer loads `pipeline.pkl` from PVC when `KSERVE_ENABLED=true`
- KServe InferenceService handles all model serving with autoscaling

Keeping them creates confusion about the actual serving path and adds unnecessary `kubectl apply` overhead in `deploy-all.sh`.

### Scope
| In Scope | Out of Scope |
|----------|-------------|
| Delete `k8s/ml/model-serving.yaml` | E2E validation (plan 57-02) |
| Delete `k8s/ml/model-serving-canary.yaml` | README update (plan 57-02) |
| Update `deploy-all.sh` Phase 34 section | model-pvc.yaml deletion (repurposed) |
| Add Phase 57 cleanup block to deploy-all.sh | KServe manifest changes |
| Verify PVC repurpose decision | Training CronJob refactoring |

### File Changes

#### 1. DELETE: `k8s/ml/model-serving.yaml`
Remove entirely. This contained:
- `Deployment/model-serving` (replicas: 0, annotated for Phase 57 deletion)
- `Service/model-serving` (ClusterIP port 8000)

Both are superseded by `KServe InferenceService/stock-model-serving` deployed via `k8s/ml/kserve/kserve-inference-service.yaml`.

#### 2. DELETE: `k8s/ml/model-serving-canary.yaml`
Remove entirely. This contained:
- `Deployment/model-serving-canary` (replicas: 0, annotated for Phase 57 deletion)

Superseded by `KServe InferenceService/stock-model-serving-canary` deployed via `k8s/ml/kserve/kserve-inference-service-canary.yaml`.

#### 3. MODIFIED: `scripts/deploy-all.sh`

**Phase 34 section** — Remove the `kubectl apply` line for `model-serving.yaml`:

```bash
# BEFORE:

# AFTER:
# (line removed — legacy model-serving.yaml deleted in Phase 57)
```

Update the echo:

**New Phase 57 section** — Insert after Phase 55 block, before Phase 5 (Kafka):
```bash
# --- Phase 57: Migration Cleanup ---
echo "[Phase 57] Verifying KServe replaces legacy serving..."
if kubectl get deployment model-serving -n ml &>/dev/null; then
    echo "[Phase 57] Removing legacy model-serving Deployment..."
    kubectl delete deployment model-serving -n ml --ignore-not-found
    kubectl delete service model-serving -n ml --ignore-not-found
fi
if kubectl get deployment model-serving-canary -n ml &>/dev/null; then
    echo "[Phase 57] Removing legacy model-serving-canary Deployment..."
    kubectl delete deployment model-serving-canary -n ml --ignore-not-found
fi
echo "[Phase 57] ✓ Legacy serving resources cleaned up"
```

#### 4. RETAIN (no change): `k8s/ml/model-pvc.yaml`
The `model-artifacts-pvc` PVC is still referenced by:
- `k8s/ml/cronjob-training.yaml` — scratch workspace during training
- `k8s/ml/cronjob-feature-store.yaml` — feature computation workspace

It is **not** used for model serving anymore (KServe pulls from MinIO S3). The PVC is repurposed, not deleted.

### Key Design Decisions
1. **Active cleanup in deploy-all.sh** — On existing clusters, stale `model-serving` Deployment/Service objects linger even after the YAML files are deleted. The Phase 57 block actively removes them for idempotent deployments.
2. **PVC retained** — Deleting the PVC would break training CronJobs. Refactoring those to write exclusively to MinIO is a separate concern (potential Phase 58+).
3. **No rollback path** — Phase 56 confirmed KServe V2 inference works end-to-end with a feature flag. The flag can still be set to `KSERVE_ENABLED=false` for legacy pickle inference without the Deployment existing — the code path doesn't require the K8s Deployment, just the pipeline.pkl file on disk.
