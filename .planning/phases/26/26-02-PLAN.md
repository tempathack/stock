## Data Strategy

The current API does not expose a dedicated SHAP or fold detail endpoint. For Phase 26:

1. **SHAP data** — Mock / sample data generator function in `src/utils/mockModelData.ts` that returns realistic SHAP values for any model name. This data shape matches what the backend SHAP analysis module produces. When a backend endpoint is added later, the mock is replaced with a `useQuery` call.

2. **Fold data** — Similarly mocked. The fold metrics structure matches the cross-validation output.

This approach delivers full UI functionality now, with a clean swap path for real data later.

## Design Decisions

- **Mock data over empty charts** — Provides full visual functionality immediately; mock data is deterministic per model name (seeded PRNG) ensuring consistent renders
- **Recharts ScatterChart for beeswarm** — True beeswarm plots require jittering which Recharts doesn't natively support; using scatter with y-axis categories approximates the SHAP beeswarm adequately
- **Blue→red gradient for feature values** — Standard SHAP color convention; cold (blue) = low feature value, hot (red) = high value
- **Dual y-axis on fold chart** — R² is on a 0–1 scale while RMSE/MAE are on a different magnitude; dual axes prevent R² from being invisible
- **Utils directory for mock data** — Keeps mock data isolated; easy to grep and remove when real API endpoints are wired
- **Grid layout for charts** — SHAP bar and beeswarm side-by-side on xl screens, stacked on smaller; fold chart always full-width below
