# Plan 17-02 — Feature Engineering & Label Generation Components

## Objective
Implement `ml/pipelines/components/feature_engineer.py` and `ml/pipelines/components/label_generator.py` — steps 2 and 3 of the Kubeflow training pipeline. Feature engineering applies all technical indicators and lag features to raw OHLCV DataFrames. Label generation creates the t+7 forward-return target with strict leakage prevention. Full test coverage. Satisfies **KF-03** and **KF-04**.

## Dependencies
- `ml/features/indicators.py` — `compute_all_indicators(df)` (Phase 10)
- `ml/features/lag_features.py` — `compute_lag_features(df)`, `compute_rolling_stats(df)`, `generate_target(df, horizon)`, `drop_incomplete_rows(df)` (Phase 11)
- `ml/pipelines/components/data_loader.py` — `load_data()` output format: `dict[str, pd.DataFrame]` (Plan 17-01)
- `ml/tests/conftest.py` — `sample_ohlcv_df` fixture (250 rows, DatetimeIndex)

## Design Decisions

1. **feature_engineer operates per-ticker:** Takes `dict[str, pd.DataFrame]`, applies indicators and lags to each DataFrame independently, returns `dict[str, pd.DataFrame]`. This prevents cross-ticker contamination in rolling/cumulative calculations (OBV, VWAP, A/D line).

2. **Column count validation:** After feature engineering, each DataFrame should have the 5 original OHLCV columns plus ~40+ indicator/lag/rolling columns. The exact count is validated in tests by running `compute_all_indicators` + `compute_lag_features` + `compute_rolling_stats` on the fixture and counting columns.

3. **label_generator also per-ticker:** The t+7 target is ticker-specific. `generate_target(df, horizon=7)` uses `shift(-7)` which is safe per-ticker. `drop_incomplete_rows()` then removes NaN rows from both the warm-up period (indicators/lags) and the tail (target).

4. **Feature names extraction:** `label_generator` returns both the data dict and a `feature_names` list. Feature names are all columns except the target column (`target_7d`). This list is passed downstream to training components.

5. **Leakage test:** A dedicated test verifies that for every row in the output, the `target_7d` value equals `(close_{t+7} - close_t) / close_t` using the **original** pre-feature-engineering close prices. This proves no information from `t+7` leaked into `t`'s features.

6. **Row count test:** After `drop_incomplete_rows()`, the output has fewer rows than the input. The exact warm-up period depends on indicator parameters (SMA-200 needs 200 rows minimum). With 250-row input, we verify the output has roughly 40-50 rows (250 - 200 warm-up - 7 target tail).

      ```python
      import logging
      import pandas as pd
      from ml.features.lag_features import generate_target, drop_incomplete_rows
      ```

      **Function:**

      ```python
      def generate_labels(
          data: dict[str, pd.DataFrame],
          horizon: int = 7,
      ) -> tuple[dict[str, pd.DataFrame], list[str]]:
      ```
      - Iterates over each `(ticker, df)` pair
      - Calls `generate_target(df.copy(), horizon=horizon)` (copy to avoid mutation)
      - Calls `drop_incomplete_rows(df)` to remove all NaN rows
      - If resulting DataFrame is empty (can happen with insufficient input rows), log warning and skip ticker
      - After processing all tickers, extract `feature_names` from the first non-empty DataFrame:
        - `feature_names = [col for col in df.columns if col != f"target_{horizon}d"]`
      - Returns `(data_dict, feature_names)`
      - Empty input → `({}, [])`
