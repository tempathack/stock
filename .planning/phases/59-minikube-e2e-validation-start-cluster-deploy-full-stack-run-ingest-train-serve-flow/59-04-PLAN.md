  <action>
    Run from /home/tempa/Desktop/priv-project/stock-prediction-platform:

    STEP A — Trigger the daily-drift CronJob:
    ```bash
    cd /home/tempa/Desktop/priv-project/stock-prediction-platform

    kubectl create job --from=cronjob/daily-drift manual-drift-59 -n ml \
      --dry-run=client -o yaml | kubectl apply -f - 2>/dev/null || \
    kubectl create job --from=cronjob/daily-drift manual-drift-59 -n ml

    echo "Waiting for drift job (up to 5 minutes)..."
    kubectl wait --for=condition=complete job/manual-drift-59 -n ml --timeout=300s
    kubectl logs job/manual-drift-59 -n ml | tail -30
    ```

    STEP B — Verify drift_logs entries in PostgreSQL:
    ```bash
    PG_POD=$(kubectl get pod -n storage -l app=postgresql -o jsonpath='{.items[0].metadata.name}')
    kubectl exec -n storage "$PG_POD" -- \
      psql -U stockuser -d stockdb \
      -c "SELECT id, ticker, drift_type, severity, detected_at FROM drift_logs ORDER BY detected_at DESC LIMIT 10;"
    ```

    If drift_logs table is empty, the drift detector may have found no drift (which is valid
    when the model is freshly trained). In that case, verify the job completed without error
    and check for JSONL output in MinIO drift-logs bucket:
    ```bash
    MINIO_POD=$(kubectl get pod -n storage -l app=minio -o jsonpath='{.items[0].metadata.name}')
    kubectl exec -n storage "$MINIO_POD" -- mc ls local/drift-logs/ 2>/dev/null \
      || kubectl exec -n storage "$MINIO_POD" -- ls /data/drift-logs/ 2>/dev/null
    ```

    A successfully completed job with 0 drift events is still a passing result (no drift
    found = system working correctly). The test requirement is that the job runs to completion
    without crashing, not that drift must be detected.

    STEP C — Run integration test for drift cycle:
    ```bash
    pytest tests/integration/test_drift_cycle.py -x -v --timeout=300 2>&1 | tail -30
    ```

    STEP D — Verify KServe rolling update behavior after drift job:
    After drift detection + auto-retrain trigger, KServe should re-read the model artifact.
    Check if the InferenceService is still Ready:
    ```bash
    kubectl get inferenceservice stock-model-serving -n ml
    kubectl get pods -n ml -l serving.kserve.io/inferenceservice=stock-model-serving
    ```
    If auto-retrain was triggered (drift severity HIGH), wait up to 10 minutes for the
    new training job to complete and KServe to reconcile:
    ```bash
    kubectl wait --for=condition=Ready inferenceservice/stock-model-serving -n ml --timeout=600s
    ```
  </action>
