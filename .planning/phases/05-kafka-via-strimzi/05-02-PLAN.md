  <action>
1. Run setup-minikube.sh to install the Strimzi operator (it also re-applies namespaces and Phase 4 resources idempotently):
   ```bash
   cd stock-prediction-platform && bash scripts/setup-minikube.sh
   ```
   Expected: Script completes with "Strimzi operator ready" and "Cluster ready" messages. The operator pod should be Running in storage namespace.

2. Verify operator is ready before proceeding:
   ```bash
   kubectl get pods -n storage -l name=strimzi-cluster-operator
   ```
   Expected: 1/1 Running.

3. Run deploy-all.sh to apply the Kafka cluster and topic manifests:
   ```bash
   cd stock-prediction-platform && bash scripts/deploy-all.sh
   ```
   Expected: Script completes with "[Phase 5] Deploying Kafka cluster and topics..." and "Deployment complete" messages.

4. Wait for the Kafka broker pod to become Ready (this takes 2-5 minutes on Minikube as KRaft initializes):
   ```bash
   kubectl wait --for=condition=Ready pod -l strimzi.io/cluster=kafka -n storage --timeout=600s
   ```
   Use 600s timeout — KRaft initialization, PVC provisioning, and metadata bootstrapping are slow on Minikube.

5. Verify all Kafka-related pods are Running:
   ```bash
   kubectl get pods -n storage
   ```
   Expected pods:
   - `strimzi-cluster-operator-*` — 1/1 Running
   - `kafka-combined-0` — 1/1 Running (the KRaft combined broker+controller)
   - `kafka-entity-operator-*` — 2/2 Running (topic operator + user operator)
   - `postgresql-*` — 1/1 Running (from Phase 4, still running)

6. Verify both KafkaTopic CRs are Ready:
   ```bash
   kubectl get kafkatopics -n storage
   ```
   Expected: Both `intraday-data` and `historical-data` topics show Ready condition.

7. Verify PVC is Bound:
   ```bash
   kubectl get pvc -n storage
   ```
   Expected: A PVC for the Kafka broker (data-kafka-combined-0) shows Bound with 10Gi capacity.

If any pod is stuck in Pending or CrashLoopBackOff:
- Check events: `kubectl describe pod <pod-name> -n storage`
- Check logs: `kubectl logs <pod-name> -n storage`
- Common issue: insufficient memory on Minikube — check `kubectl describe node minikube` for resource pressure
- If metadataVersion 3.7-IV4 causes issues, try removing the metadataVersion field entirely (let Strimzi default it)
