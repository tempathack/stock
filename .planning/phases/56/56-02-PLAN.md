## Plan 56-02 — A/B Testing via KServe Canary & Backward-Compatible Responses

### Objective
Migrate A/B model testing from application-level weighted routing (loading different `pipeline.pkl` files) to KServe infrastructure-level traffic splitting, where the API routes requests to primary or canary KServe InferenceService URLs.

### Requirements Addressed
- **KSERV-12**: A/B model testing uses KServe canary traffic splitting (percentTraffic)
- **KSERV-11**: Prediction API response schema backward-compatible (verified end-to-end)

### Why
Phase 49 implemented A/B testing with application-level routing — the API pod itself loaded different model pipelines and routed traffic via weighted random. With KServe, models are served externally, so the A/B split should leverage KServe's native canary support. This provides:
- Infrastructure-level traffic splitting (more reliable than app-level randomization)
- Independent scaling of primary and canary model servers
- KServe handles health checking and automatic rollback on canary failure
- Cleaner separation: API pod handles business logic, KServe handles model serving

### Scope
| In Scope | Out of Scope |
|----------|-------------|
| Canary InferenceService YAML | Primary InferenceService (phase 55) |
| ab_service.py KServe URL routing | KServe V2 client (plan 56-01) |
| Config: KSERVE_CANARY_URL | Frontend changes (none needed) |
| model-serving-canary.yaml scale to 0 | Legacy manifest removal (phase 57) |
| prediction_service kserve_url passthrough | Model training pipeline changes |

### Key Design Decisions
1. **Separate InferenceService CR** — canary is a distinct InferenceService (not InferenceService canaryTrafficPercent) because the API controls routing at the application level. This preserves the existing A/B logging and `assigned_model_id` tracking that the frontend reads.
2. **Application-level routing preserved** — while KServe supports native `canaryTrafficPercent`, we route at the API level so we can log which model served which request (A/B assignment tracking). KServe-native canary is opaque to the app.
3. **Two KServe Services, not one with canary spec** — allows independent scaling, independent model artifacts, and explicit control over which requests go where.
4. **Feature flag chain** — `AB_TESTING_ENABLED` AND `KSERVE_ENABLED` must both be True for KServe canary routing. If either is False, behavior degrades gracefully.
